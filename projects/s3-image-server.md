# AWS S3 + CloudFront 이미지 서버 (Kotlin Coroutine)

> 기간: 2024.09 ~ 2024.10
> 기술: Kotlin(Coroutine), Java, Spring Boot, AWS S3 / CloudFront

## 📌 문제
OMS와 동일한 서버에 이미지가 저장되어 있어 서버 자원과 인프라 비용 부담이 컸다.
이를 S3 + CloudFront 환경으로 분리·이관해야 했다.
또한 이 업로드 API는 이관 후에도 **상시 운영되는 기능**으로,
여러 시설에서 동시에 대량의 이미지를 업로드하는 환경이었다.

## ⚙️ 해결 — 왜 Kotlin Coroutine을 선택했나

대량 이미지 업로드는 **I/O 바운드 작업**이다. 만약 스레드를 블로킹하는 방식으로
처리하면, 동시 요청이 몰릴 때 스레드 풀이 빠르게 고갈되고 I/O 에러가 발생할 수 있다.

이를 막기 위해 **Kotlin Coroutine의 논블로킹 비동기 처리**로 업로드 서버를 설계했다.
- 적은 스레드로 많은 동시 I/O 요청을 처리할 수 있어 스레드 고갈 위험을 줄임
- 콜백 중첩 없이 동기 코드처럼 읽혀 유지보수성이 좋음
- 동시 다발 업로드 상황에서도 안정적으로 부하를 견딤

## 📈 결과
- 이미지 로딩 속도 약 **2배 개선(실측)**
- 인프라 비용 절감 (이미지 분리로 서버 자원 부담 감소)
- 프로모션 등 트래픽 집중 시 이미지가 표시되지 않던 문제 완전 해소

## 💡 비고
"왜 비동기인가"가 핵심이다. 단순히 코루틴을 써본 것이 아니라,
**동시 다발 I/O → 블로킹 시 스레드 고갈 위험 → 논블로킹 선택**이라는
판단 과정을 거쳐 기술을 선택했다.
