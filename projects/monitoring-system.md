# 모니터링 · 알림 체계 구축

> 기간: 2025.03 ~ 2025.04
> 기술: Prometheus, Grafana, Sentry

## 📌 문제
PHP/Java가 혼재된 환경에서 장애 원인 추적이 어려워 대응이 지연되었다.
또한 단순 알림만으로는 "운영이 처리할 일"과 "개발이 처리할 일"이 구분되지 않아
불필요한 알림 피로가 컸다.

## ⚙️ 해결
- PHP는 Sentry, Java는 Prometheus + Grafana / Sentry로 모니터링 구성
- **에러 레벨·발생 주기를 튜닝**하여 노이즈성 알림 정리
  - 예: 3~5분 주기로 발생하던 에러 중, 타 채널 오설정 등 운영이 처리할 건을 분리
- **처리 주체별 알림 라우팅 설계**
  - 운영팀이 처리할 에러 → Google Chat
  - 개발자가 처리할 에러 → Discord

## 📈 결과
- 장애 탐지·대응 속도 단축, 실시간 대응 체계 확립
- 처리 주체가 명확해져 알림 피로 감소 및 대응 효율 향상

## 💡 비고
이 프로젝트의 핵심은 도구 도입 자체보다, **에러를 처리 주체별로 분류해
알림을 라우팅한 설계**에 있다. 누가 무엇을 처리해야 하는지를 알림 단계에서
구분함으로써 운영-개발 간 대응 프로세스를 정비했다.
